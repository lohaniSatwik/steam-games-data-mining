Abstract. This project applies data mining and machine learning techniques to a dataset of approximately 120,000 Steam games sourced from Kaggle. We pursue three complementary analytical tasks: (1) a binary classification task predicting whether a game will receive a positive overall review outcome, (2) a regression task estimating peak concurrent users as a proxy for commercial success, and (3) an unsupervised clustering task to discover natural game market segments. Each task employs symbolic models, non-symbolic models, and a naive baseline, evaluated under 10-fold stratified cross-validation. Results will be compared against publicly available Kaggle solutions.

1. Business Understanding

The Steam marketplace hosts over 50,000 active titles, making it one of the most competitive digital distribution platforms in the world. Game developers and publishers routinely face high-stakes decisions about genre, pricing, and release timing with limited quantitative guidance. This project addresses three concrete business questions derived from the available data:

  • Reception prediction: Can we predict, from a game's attributes (genre, price, developer history, tags), whether it will receive a predominantly positive or negative review outcome?
  • Popularity estimation: Can we estimate peak concurrent user counts — a measure of commercial reach — from observable game features?
  • Market segmentation: What natural clusters of games exist in the Steam catalogue, and how do these segments differ in price, playtime, and audience size?

Answers to these questions can inform data-driven publishing strategy, competitive positioning, and pricing decisions for new game releases.

2. Dataset Description

The dataset used is the Steam Games Dataset published by FronkonGames on Kaggle (fronkongames/steam-games-dataset), last updated in 2024. It contains metadata for approximately 120,000 Steam game entries retrieved via the Steam API and SteamSpy.

Key Features Available
  • Price (USD), release date, developer, publisher
  • Genres, categories, user-defined tags (multi-value)
  • Positive and negative review counts
  • Peak concurrent users (CCU), average playtime
  • Estimated owner range (e.g., "50,000–100,000")
  • Supported platforms, languages, age rating
Known Data Challenges
  • Owner counts stored as string ranges, not exact numbers
  • Genres and tags stored as JSON lists — require parsing
  • Large class imbalance: majority of games have zero or very few reviews
  • Heavy right skew in CCU (a few titles dominate)
  • Many free-to-play games with atypical price distributions

Since this is a Kaggle dataset, results will be compared against publicly available kernels and notebooks, as required by the IE500 project guidelines.

3. Exploratory Data Analysis

Prior to modelling, we will conduct a structured exploratory analysis to characterise the dataset and motivate preprocessing decisions. Planned analyses include:

  • Distribution of review ratios (positive / total) and class balance assessment
  • Price distribution stratified by free-to-play vs. paid games
  • Frequency distribution of top-50 genres and user-defined tags
  • Scatter plots of price vs. review ratio, and log-CCU vs. playtime
  • Pearson correlation heatmap across all numeric features
  • Identification of outliers in CCU and playtime using box plots and IQR method

4. Data Preprocessing and Feature Engineering

All preprocessing steps will be applied consistently across all three tasks. The cleaned dataset will be exported to Parquet format to ensure reproducibility.

Cleaning Steps
  • Remove games with zero total reviews (no ground truth available for supervised tasks)
  • Parse estimated owner ranges to numeric midpoints (e.g., "50k–100k" → 75,000)
  • Explode multi-value genre and tag lists into binary indicator columns using Multi-Label Binarizer
  • Apply median imputation for missing prices (DLC / bundle anomalies)
  • Cap or flag extreme CCU outliers (e.g., Dota 2, CS:GO) to limit leverage
Feature Engineering
  • Derive review_ratio = positive reviews / total reviews
  • Binarise review_ratio at threshold 0.70 to produce classification target label
  • Log-transform peak_ccu and price to reduce right skew for regression
  • Bin release year into era buckets (pre-2010, 2010–2015, 2016–2019, 2020+)
  • Retain top-50 tags by frequency to control feature dimensionality
  • Standardise all continuous features using StandardScaler

5. Task 1 — Classification: Predicting Review Outcome

5.1 Task Formulation

The target variable is a binary label derived from review_ratio: Positive (≥ 70% positive reviews) and Negative (< 70%). This threshold is consistent with Steam's own "Mostly Positive" boundary. Class imbalance will be addressed using SMOTE oversampling on the training folds and the class_weight='balanced' parameter where applicable.

5.2 Models

Model Type Justification
Majority Class Classifier Baseline Always predicts the dominant class. Establishes the minimum performance floor that all other models must exceed.
Logistic Regression (L2) Symbolic Linear, interpretable model. Coefficient magnitudes reveal which features most influence review outcome.
Random Forest Non-Symbolic Ensemble of decision trees. Handles non-linear interactions between genre, price, and tags. Provides feature importances.
XGBoost Non-Symbolic Gradient-boosted trees. Expected to achieve highest accuracy; hyperparameter tuning via nested CV.

5.3 Evaluation Metrics

Primary metric: macro F1-score (accounts for class imbalance across both labels). Secondary metric: AUC-ROC (measures discriminative ability irrespective of threshold). Both metrics reported as mean ± standard deviation across 10 folds.

6. Task 2 — Regression: Estimating Peak Concurrent Users

6.1 Task Formulation

The target variable is log₁₀(peak_ccu + 1), the log-transformed peak concurrent user count. Log-transformation is applied because CCU is heavily right-skewed: a handful of titles (e.g., Dota 2, CS:GO) have orders-of-magnitude more players than the median game. The log scale compresses this range and makes linear relationships more learnable. Predictions will be back-transformed to the original scale for interpretation.

6.2 Models

Model Type Justification
Mean Value Regressor Baseline Always predicts the training set mean. Provides a naive RMSE benchmark that all models must beat.
Ridge Regression (L2)
Lasso Regression (L1)		 Symbolic Linear models with regularisation. Lasso performs implicit feature selection by shrinking irrelevant coefficients to zero. Interpretable coefficients reveal which features drive CCU.
Gradient Boosting Regressor Non-Symbolic Sequential ensemble model that iteratively corrects residuals. Captures complex non-linear relationships between genre, price, and player counts.

6.3 Evaluation Metrics

Primary metric: RMSE (root mean squared error, on the log scale). Secondary metrics: MAE and R². Residual plots (predicted vs. actual log-CCU) will be analysed to identify systematic biases.

7. Task 3 — Clustering: Game Market Segmentation

7.1 Task Formulation

This unsupervised task aims to discover natural groupings of games based on their commercial and gameplay characteristics, without using any predefined labels. Input features will include: log-price, log-CCU, average playtime, review_ratio, and binary genre/tag indicators (top-30 by frequency). All features will be standardised prior to clustering.

7.2 Methods

K-Means (MiniBatch variant)
  • Partition-based algorithm; computationally efficient on 100k+ records
  • Optimal k selected via elbow method and silhouette analysis (range: k = 3–12)
DBSCAN
  • Density-based algorithm; does not require specifying k in advance
  • Naturally identifies noise points (games that belong to no cluster)
  • Parameters ε and minPts tuned via k-distance graph

7.3 Evaluation and Interpretation

Internal validation metrics: Silhouette score and Davies-Bouldin index. Cluster profiles will be generated by computing the mean value of each feature per cluster, and descriptive segment labels assigned (e.g., "Free-to-Play Competitive", "Premium Indie", "Casual Puzzle"). Clusters will be visualised in 2D using PCA projection.

8. Evaluation Methodology

8.1 Cross-Validation Strategy

All supervised models (Tasks 1 and 2) will be evaluated using 10-fold stratified cross-validation. Stratification ensures that the class ratio (for classification) or the CCU distribution (for regression, via quantile-based stratification) is preserved across all folds. No test data is ever used during training or tuning.

8.2 Hyperparameter Optimisation

Hyperparameter selection will follow a nested cross-validation procedure to avoid overfitting to the validation set. The outer loop (5-fold) estimates generalisation performance; the inner loop (3-fold grid/random search) selects the best hyperparameter configuration. Key parameters to tune:

  • Random Forest / XGBoost: n_estimators, max_depth, min_samples_leaf, learning rate (XGB), subsample ratio
  • Ridge / Lasso: regularisation strength α (log-scale grid: 10⁻³ to 10³)
  • Gradient Boosting: n_estimators, max_depth, learning rate, min_samples_split

8.3 State-of-the-Art Comparison Required — Kaggle Dataset

Because the dataset is published on Kaggle, the IE500 guidelines require that results be compared against publicly available solutions. We will identify 2–3 reference notebooks from Kaggle's public kernel section for this dataset and compare our F1, RMSE, and R² scores against their reported values. Differences will be explained in terms of feature engineering choices, model complexity, and dataset filtering decisions.

9. Expected Results

  • Classification: We expect XGBoost to outperform Logistic Regression and both to clearly exceed the majority-class baseline. Genre and review_ratio-derived features are hypothesised to be the strongest predictors.
  • Regression: CCU prediction is inherently noisy; we expect moderate R² (0.35–0.55) given the stochastic nature of game popularity. Gradient Boosting is expected to outperform Ridge/Lasso.
  • Clustering: We expect 4–7 meaningful clusters aligned with broad market tiers (e.g., AAA paid, free-to-play competitive, budget indie, narrative/niche). DBSCAN is expected to isolate viral outlier games as noise points.
